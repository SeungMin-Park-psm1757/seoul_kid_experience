# 한계 및 레드팀 체크

## 핵심 한계

- 현재 샘플 데이터는 실제 운영 정보가 아니라 제출 구조 검증용 데이터다.
- 이동시간은 실제 교통 API가 아니라 자치구 간 상대거리 기반 추정 범위다.
- 예약 가능 여부는 실시간 잔여석을 의미하지 않는다.
- 원문 링크, 취소 규칙, 노쇼 규칙, 준비물은 데이터셋별 품질 차이가 크다.
- 좌표가 없는 프로그램은 지도에서 누락될 수 있다.
- 날짜가 행사일인지 접수일인지 원본 데이터셋마다 다를 수 있어 매핑 검수가 필요하다.

## 금지 표현

- "예약 완료"
- "실시간 잔여석 보장"
- "정확한 이동시간"
- "자동 신청"
- "가장 좋은 프로그램"

## 권장 표현

- "원문 링크에서 최종 확인"
- "자치구 간 상대거리 기반 추정"
- "데이터 기반 탐색 결과"
- "추천 이유"
- "조건을 완화하면 가능한 후보"

## 레드팀 체크리스트

- 샘플 데이터인데 실제 운영 서비스처럼 보이지 않는가?
- 접수 예정 후보가 예약 가능 후보보다 위에 있지 않은가?
- 30일 초과 미래 후보가 기본 추천/비교/플랜B 기본값에 섞이지 않는가?
- 이동시간이 정확한 교통시간처럼 보이지 않는가?
- 원문 링크가 없을 때 확인 필요로 표시되는가?
- 정책 차트가 결론 없이 그림만 나열되지 않는가?
- 데이터 품질 패널에서 샘플 데이터 여부를 명확히 알 수 있는가?

## 제출 전 필수 확인

- 실제 서울 공공데이터로 교체 후 `docs/validation_report.md`를 다시 작성한다.
- 날짜 필드가 행사일인지 접수일인지 출처별로 확인한다.
- 예약상태 원문값을 상태 우선순위에 맞게 정규화한다.
- PPT에는 샘플/실제 데이터 여부와 기준일을 명시한다.
