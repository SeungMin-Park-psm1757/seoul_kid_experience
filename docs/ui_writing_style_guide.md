# UI 문구 작성 가이드

작성일: 2026-05-13

## 1. 부모용 화면은 일상어 사용

부모용 홈, 추천 탐색, 후보 비교, 상세 화면에서는 행정·정책·개발 용어를 피한다. “하드필터”, “저마찰”, “데이터 품질 패널” 같은 말은 쓰지 않는다.

## 2. 정책 화면은 분석어 사용 가능

서울시 관점 탭에서는 선택가능도, 이용장벽, 우선 점검, 자치구 유형화 같은 분석어를 사용할 수 있다. 단, 특정 지역을 낙인찍는 표현은 피한다.

## 3. 예약 가능성은 확정적으로 말하지 않음

`접수중`, `예약가능`, `상시이용` 상태도 신청 성공을 보장하지 않는다. “예약 확정”, “잔여석 확정”, “바로 예약 완료” 같은 표현은 금지한다.

## 4. 공식 링크에서 확인 원칙 유지

모든 추천 카드와 주의 문구는 최종 신청 가능 여부를 공식 예약/출처 링크에서 확인해야 한다는 원칙을 유지한다.

## 5. 큰 데이터 수치를 첫 화면에서 강조하지 않음

첫 화면은 “후보가 몇백 개 있다”가 아니라 “오늘 먼저 볼 후보는 무엇인가”를 보여준다. 큰 수치는 더보기 또는 데이터 확인 영역에 둔다.

## 6. 추천 이유는 1~2문장으로 짧게

추천 이유는 나열식 점수 설명이 아니라 부모가 바로 이해할 수 있는 문장이어야 한다.

좋은 예:

- “이번 주말 일정이고, 6~13세 대상·무료·실내 조건에 맞습니다.”
- “첫 후보가 어렵다면 같은 날짜권에서 확인해볼 만한 대체 후보입니다.”

## 7. 주의 문구는 불안을 키우지 않고 정확하게

주의 문구는 무섭게 쓰지 않는다. 단, 데이터 한계와 공식 확인 필요성은 숨기지 않는다.

## 8. 버튼은 동사형으로

버튼은 “무엇을 하게 되는지”가 보여야 한다.

- 좋은 예: `추천 결과 보기`, `상세 보기`, `대체 후보 보기`
- 검토 필요: `전체 지도`, `대체 후보`

## 9. 같은 개념은 같은 용어로

동일 개념은 화면 전체에서 같은 표현을 쓴다.

- 플랜B 계열: `대체 후보`
- 공식 확인 계열: `공식 링크에서 확인`
- 제외 계열: `지금은 추천하지 않은 이유`

## 10. 내부 개발 용어는 사용자 화면에 노출하지 않음

내부 구현과 테스트에는 `hard_filter`, `date_match_level`, `reservation_friction` 같은 용어를 사용할 수 있지만 사용자 화면에는 직접 노출하지 않는다.

## 금지 표현

- 예약 확정
- 잔여석 확정
- 자동예약
- 예약 완료
- 하드필터
- 저마찰
- 조건 완화 퍼널
- 데이터 품질 패널
- API 확인
- 링크 확인 필요

## 권장 표현

- 공식 링크에서 확인
- 조건에 맞는 후보
- 신청 부담 낮음
- 조건을 넓히면 늘어나는 후보
- 데이터 확인
- 지금은 추천하지 않은 이유
