# 부모용 MVP 시각화 기획 상세 요약

## 1. 기획 방향 전환

기존 MVP는 서울시 관점의 정책 시각화에 무게가 있었고, 사용자가 첫 화면에서 바로 "이번 주말 아이를 어디 보낼지" 판단하기 어려웠다. 수정된 방향은 1차 타깃을 부모 사용자로 두고, 정책 인사이트는 별도 탭으로 후순위 배치하는 것이다.

핵심 질문은 다음 하나로 압축한다.

> 이번 주말, 우리 아이 어디 갈까?

이 앱은 예약 서비스나 회원 기반 서비스가 아니라, 부모가 주말 아이체험 후보를 빠르게 좁혀보는 인터랙티브 데이터 시각화 결과물이다. 따라서 로그인, 회원가입, 자동예약, 알림, DB 저장은 구현하지 않는다. 최종 예약 가능 여부와 상세 규정 확인은 각 프로그램의 출처 링크에서 하도록 설계한다.

## 2. 핵심 사용자와 사용 맥락

주 사용자는 주말에 아이와 갈 체험 프로그램을 찾는 보호자다. 사용자는 긴 정책 지표보다 "우리 아이에게 맞는가", "이번 주말 가능한가", "멀지 않은가", "비용이 부담되지 않는가", "실내인가", "예약 가능한가"를 먼저 확인하려 한다.

사용 상황은 다음과 같이 가정한다.

- 금요일 저녁이나 토요일 오전에 빠르게 후보를 고른다.
- 아이 나이와 출발 자치구를 먼저 정한다.
- 너무 멀거나 비용이 높은 후보는 빠르게 제외한다.
- 비가 오거나 날씨가 불안하면 실내 후보를 우선 본다.
- 첫 후보가 마감일 경우 바로 플랜B를 확인하고 싶어 한다.
- 정책적 의미는 나중에 확인하거나 발표 자료의 후반부에서 본다.

## 3. MVP 목표

부모가 30초 안에 갈 곳 후보 2~3개를 고를 수 있게 하는 것이 핵심 목표다. 첫 화면은 정책 지도나 지표가 아니라 추천 가능한 프로그램 카드가 되어야 한다.

구체 목표는 다음과 같다.

- 아이 나이, 학년, 출발 자치구, 날짜, 이동시간, 비용, 실내/실외, 관심분야를 입력하거나 필터링할 수 있게 한다.
- 결과는 정책지표가 아니라 프로그램 카드로 먼저 보여준다.
- 카드에는 의사결정에 필요한 핵심 정보를 한눈에 배치한다.
- 후보 2~3개를 비교할 수 있는 화면을 제공한다.
- 상세 화면에서는 보호자 동반 여부, 준비물, 취소/노쇼 규칙, 주변 대체안, 유사 프로그램을 보여준다.
- 기존 정책 시각화는 삭제하지 않고 "서울시 관점 / 정책 인사이트" 탭으로 이동한다.

## 4. 전체 화면 구조

앱은 다음 5개 탭으로 재구성한다.

1. 부모용 홈
2. 추천 탐색
3. 후보 비교
4. 상세 + 플랜B
5. 정책 인사이트

이 순서는 사용자의 실제 의사결정 순서를 따른다. 먼저 후보를 받고, 더 탐색하고, 비교하고, 상세 규칙과 대체안을 확인한 뒤, 마지막에 정책 관점의 시각화를 본다.

## 5. 부모용 홈

### 목적

첫 화면에서 "이번 주말 우리 아이 어디 갈까?"라는 질문에 즉시 답하는 화면이다. 사용자는 복잡한 지표를 보지 않고, 핵심 필터와 추천 카드만으로 후보를 빠르게 좁힌다.

### 제목과 설명 문구

제목은 다음과 같이 설정한다.

> 이번 주말, 우리 아이 어디 갈까?

설명 문구는 정책지도 중심이 아니라 부모용 선택 지원 도구로 표현한다.

> 부모가 주말 아이체험 후보를 30초 안에 좁혀볼 수 있도록 돕는 데이터 기반 시각화입니다. 정책 지표는 뒤쪽 탭에 두고, 첫 화면은 나이·거리·비용·실내 여부·관심분야에 맞는 프로그램 카드부터 보여줍니다.

### 핵심 필터

첫 화면 필터는 빠른 선택을 위해 한 번에 보이는 형태로 구성한다.

- 아이 나이: 4세부터 12세까지 선택
- 학년/단계: 미취학, 초1~초6
- 출발 자치구: 서울 25개 자치구 중 선택
- 날짜: 기본값은 다음 토요일
- 이동시간: 15~75분 범위 슬라이더
- 비용: 무료, 저비용, 유료
- 실내/실외: 실내, 실외, 혼합
- 관심분야: 과학, 미술, 자연, 놀이, 독서, 체육 등 데이터 내 카테고리

### 추천 요약

필터 적용 후 사용자가 빠르게 상황을 이해하도록 4개의 요약 카드를 배치한다.

- 추천 가능 후보 수
- 가장 가까운 후보의 예상 이동시간
- 무료/저비용 후보 수
- 날짜 기준: 선택 날짜 일치 여부 또는 전후 30일 후보 포함 여부

### 추천 카드

추천 카드 3~6개를 우선 보여준다. 카드의 목적은 사용자가 한눈에 "갈 수 있는지" 판단하게 하는 것이다.

각 카드에 포함되는 정보는 다음과 같다.

- 프로그램명
- 자치구
- 장소
- 대상연령
- 날짜/시간
- 비용
- 실내/실외
- 예약상태
- 예상 이동시간
- 이용규칙 요약
- 예약/출처 링크
- 상세 보기 버튼
- 플랜B 보기 버튼

카드는 정책 점수보다 실제 선택 요소를 먼저 보여준다. 예약상태는 시각적으로 강조하고, 무료/저비용, 실내 여부, 이동시간은 배지 형태로 빠르게 읽히게 한다.

## 6. 추천 탐색

### 목적

홈에서 제시된 추천 외에 사용자가 더 넓게 후보를 탐색하는 화면이다. 지도와 리스트를 함께 제공해 위치감과 선택지를 동시에 확인한다.

### 구성

- 지도: 후보 프로그램의 위도/경도 기반 위치 표시
- 리스트: 추천 후보 카드 목록
- 정렬 옵션
- 동일 필터 결과를 기반으로 한 탐색

### 정렬 기준

정렬은 부모의 실제 판단 기준에 맞춘다.

- 가까운 순: 예상 이동시간이 짧은 후보 우선
- 무료 우선: 무료 또는 저비용 후보 우선
- 마감임박: 선택 날짜와 가까운 후보 우선
- 실내 우선: 실내 또는 혼합 후보 우선

### 지도 표현

지도는 정책 경계 지도가 아니라 프로그램 위치를 보여주는 탐색 지도다. 마커는 카테고리 색상으로 구분하고, hover 정보에는 프로그램명, 자치구, 장소, 카테고리, 예약상태, 예상 이동시간을 표시한다.

## 7. 후보 비교

### 목적

부모가 최종 후보 2~3개를 놓고 빠르게 비교하도록 돕는다. 이 화면은 "무엇이 더 나은가"를 표로 정리하는 의사결정 보드다.

### 비교 대상

추천 상위 후보 중 최대 3개를 선택한다. 기본값은 추천 상위 3개다.

### 비교 항목

비교표에는 다음 항목을 넣는다.

- 프로그램명
- 대상연령
- 거리/예상 이동시간
- 비용
- 날짜/시간
- 예약상태
- 실내/실외
- 이용규칙 요약
- 플랜B 힌트

### 해석 방식

비교표는 점수표가 아니라 선택표다. 부모가 "가장 가까운 것", "무료인 것", "규칙이 덜 복잡한 것", "비가 와도 가능한 것"을 직접 비교하게 한다.

## 8. 상세 + 플랜B

### 목적

최종 후보를 정하기 전 세부 규칙과 대체안을 확인하는 화면이다. 홈 카드가 빠른 판단이라면, 상세 화면은 실제 행동 직전 확인 단계다.

### 상세 정보

상세 화면은 선택한 프로그램의 카드를 다시 보여주고, 아래 정보를 추가한다.

- 보호자 동반 여부
- 준비물
- 취소 규칙
- 노쇼 규칙
- 이용규칙 요약
- 예약/출처 링크

현재 데이터에 준비물 전용 컬럼이 없으므로 MVP에서는 "물, 간식, 편한 복장 권장. 기관별 준비물은 예약 페이지에서 최종 확인"으로 안내한다. 실제 데이터 확장 시 준비물 컬럼을 별도 수집해 대체할 수 있다.

### 플랜B

플랜B는 첫 후보가 마감되거나 조건이 맞지 않을 때 바로 볼 수 있는 주변/유사 대체안이다.

대체안 기준은 다음과 같다.

- 같은 자치구 후보
- 같은 카테고리 후보
- 현재 후보보다 이동시간이 크게 늘지 않는 후보
- 아이 나이에 맞는 후보
- 예약 가능 또는 상시 운영 후보

플랜B는 서비스형 자동예약이 아니라 데이터 기반 대안 탐색이다. 사용자는 대체안의 출처 링크를 직접 열어 최종 예약 여부를 확인한다.

## 9. 정책 인사이트

### 목적

기존 정책 시각화의 장점을 유지하되 첫 화면에서 뒤로 물린다. 이 탭은 부모의 즉시 선택보다는 발표, 정책 검토, 지역 격차 해석을 위한 후반부 화면이다.

### 포함 시각화

다음 기존 시각화를 유지한다.

- 빈 주말 캘린더
- 플랜B 충분도 지도
- 연령별 블라인드 스팟
- 선택가능도 x 플랜B 매트릭스
- 자치구 유형화
- 이용장벽 차트

### 정책 질문

정책 인사이트 탭에서 다루는 질문은 다음과 같다.

- 빈 주말이 반복되는 자치구는 어디인가?
- 첫 선택이 마감되었을 때 인접 자치구까지 확장해도 플랜B가 부족한 곳은 어디인가?
- 특정 연령대에서 선택가능도가 급감하는 자치구는 어디인가?
- 비용, 예약 의존, 연령 제한, 이용규칙 중 어떤 요소가 선택장벽으로 작동하는가?

## 10. 데이터 필드 매핑

부모용 카드와 상세 화면은 `data/processed/unified_programs.csv`의 프로그램 단위 데이터를 사용한다.

주요 필드 매핑은 다음과 같다.

- `title`: 프로그램명
- `category`: 관심분야
- `district`: 자치구
- `venue`: 장소
- `age_min`, `age_max`, `age_label`: 대상연령
- `fee`, `fee_type`: 비용
- `indoor_outdoor`: 실내/실외
- `date_start`, `date_end`, `time_text`: 날짜/시간
- `status_label`: 예약상태
- `reservation_required`: 예약 필요 여부
- `rule_summary`: 이용규칙 요약
- `no_show_rule`: 노쇼 규칙
- `cancel_rule`: 취소 규칙
- `family_fit`: 보호자 동반 여부 또는 가족 적합성
- `lat`, `lng`: 지도 표시 위치
- `source_url`: 예약/출처 링크

정책 인사이트 탭은 기존 지표 파일을 사용한다.

- `data/processed/district_age_metrics.csv`: 자치구·연령대별 선택가능도, 플랜B, 이용장벽, 유형화
- `data/processed/weekly_choice_metrics.csv`: 주차별 선택가능도와 빈 주말 탐색

## 11. 추천 로직

추천 로직은 복잡한 개인화 모델이 아니라 MVP용 탐색 점수다. 사용자가 입력한 필터를 먼저 적용한 뒤, 남은 후보를 추천 점수로 정렬한다.

### 필터 조건

필수 필터는 다음과 같다.

- 아이 나이가 프로그램 대상연령에 포함되어야 한다.
- 예약 가능, 상시 운영, 접수 예정 상태를 우선한다.
- 예상 이동시간이 사용자가 설정한 범위 이내여야 한다.
- 비용 조건과 실내/실외 조건이 맞아야 한다.
- 관심분야 조건이 맞아야 한다.

### 날짜 처리

선택 날짜와 일치하는 후보가 있으면 해당 후보를 우선 보여준다. 정확히 일치하는 후보가 없으면 선택 날짜 전후 30일 내 후보를 포함해 사용자가 빈 화면을 보지 않도록 한다.

### 추천 점수 구성

추천 점수는 다음 요소를 결합한다.

- 예약 가능성
- 연령 적합성
- 선택 날짜 일치 여부
- 무료 여부
- 실내/혼합 여부
- 예상 이동시간
- 선택 날짜와의 거리

이 점수는 사용자를 대신해 결정을 내리는 자동 추천이 아니라, 부모가 빠르게 후보를 정렬해 보기 위한 탐색용 점수다.

## 12. 이동시간 처리

현재 MVP에는 실제 대중교통/도보 경로 API를 연결하지 않는다. 대신 출발 자치구와 프로그램 자치구의 상대 위치 차이를 이용해 예상 이동시간을 계산한다.

이동시간은 다음처럼 안내한다.

> 이동시간은 자치구 간 상대거리 기반 추정값입니다. 실제 예약 가능 여부는 출처 링크에서 최종 확인하세요.

향후 확장 시에는 서울시 대중교통 API, 지도 API, 또는 실제 좌표 기반 거리 계산으로 대체할 수 있다.

## 13. 시각 디자인 방향

디자인은 행정 대시보드보다 부모용 큐레이션 보드에 가깝게 구성한다. 다만 "서비스 개발처럼 보이기보다 인터랙티브 시각화 결과물"이라는 요구를 유지하기 위해 과도한 로그인/예약 UI나 커머스형 CTA는 배제한다.

시각 톤은 다음을 따른다.

- 따뜻한 베이지 배경
- 카드형 정보 구조
- 배지로 핵심 속성 강조
- 정책 지표보다 프로그램명을 크게 배치
- 지도와 표는 탐색 보조로 사용
- 예약 링크는 외부 출처 확인용으로만 제공

## 14. 구현하지 않는 범위

이번 MVP에서 명시적으로 제외하는 기능은 다음과 같다.

- 로그인
- 회원가입
- 자동예약
- 알림
- DB 저장
- 사용자별 즐겨찾기 저장
- 실제 결제
- 실시간 잔여석 보장
- 자동 경로 탐색

이 제외 범위는 기획상 중요하다. 앱의 성격을 예약 서비스가 아니라 데이터 기반 의사결정 시각화로 유지하기 위한 장치다.

## 15. 발표/포트폴리오 관점에서의 메시지

이 시각화의 메시지는 "서울의 공공 아이체험 데이터를 부모의 실제 선택 언어로 재구성했다"는 것이다.

발표 흐름은 다음이 적합하다.

1. 기존 정책 지표만으로는 부모가 바로 행동하기 어렵다는 문제 제기
2. 부모의 실제 선택 기준을 필터로 전환
3. 프로그램 카드를 통해 30초 안에 후보를 고르는 경험 제시
4. 비교와 플랜B로 실제 주말 의사결정 흐름 완성
5. 마지막에 정책 인사이트 탭으로 지역 격차와 공급 문제 해석

## 16. 향후 개선 제안

다음 단계에서는 데이터 정확도와 실제 사용성을 높일 수 있다.

- 실제 예약 가능 여부 업데이트 주기 표시
- 준비물 컬럼 별도 수집
- 보호자 동반 필수/권장/불필요 구분 정교화
- 자치구 간 이동시간을 실제 대중교통 시간으로 교체
- 날씨 조건에 따른 실내 우선 추천
- 프로그램 상세 페이지의 원문 규칙 파싱
- 플랜B를 같은 날짜, 같은 시간대 기준으로 더 엄격하게 추천
- 발표용 캡처 화면과 PPT용 요약 이미지 생성

## 17. 현재 구현 상태 요약

현재 앱은 다음 상태까지 구현되어 있다.

- 첫 화면 제목과 설명을 부모용으로 변경
- 핵심 필터 제공
- 추천 요약 카드 제공
- 프로그램 카드 3~6개 우선 노출
- 추천 탐색 탭에서 지도 + 리스트 제공
- 가까운 순, 무료 우선, 마감임박, 실내 우선 정렬 제공
- 후보 비교 탭에서 최대 3개 비교표 제공
- 상세 + 플랜B 탭에서 규칙, 링크, 대체안 제공
- 기존 정책 시각화는 정책 인사이트 탭으로 이동

이 구조는 부모의 빠른 선택과 정책적 해석을 모두 담되, 우선순위는 부모의 실제 주말 의사결정에 둔다.
